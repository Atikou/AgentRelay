# 项目整体架构

本文给出 **AgentRelay** 项目的整体架构视图：分层设计、模块职责、关键调用链路、目录结构与里程碑路线图。新人或其他 Agent 可据此快速建立全局认知，再深入到各专题文档。

> 设计类文档（能力清单 `agent-todolist.md`、实现指南 `Agent_TS_实现指南_修订版.md`）在仓库根目录；本架构文档随代码演进持续更新。

## 一、分层总览

## 二、模块职责

| 层 | 模块 | 路径 | 职责 |
| --- | --- | --- | --- |
| 交互 | 测试台网页 | `agent-relay/public/` | 无框架 HTML/CSS/JS，验证各能力的按钮面板；Agent 结果卡展示 `executionMeta.workflowType` 对应的内部处理状态 |
| 交互 | CLI | `agent-relay/src/cli/` | `main.ts` 框架自检、`check-models.ts` 连通性诊断 |
| 服务 | HTTP 服务 | `agent-relay/src/server/` | `createHttpServer` + `handlers/*` 按域路由；`server.ts` 仅入口 |
| 应用 | AppContext | `agent-relay/src/app/` | 依赖容器：模型/工具/上下文/调度/编排单例装配 |
| 编排 | Orchestrator | `agent-relay/src/orchestrator/Orchestrator.ts` | 统一 Run：Agent/Task/Plan/Scheduler；单次 Chat 委派给 ChatService；`POST /api/agent/resume` 注入 `RunState.location` 续跑定位 |
| 编排 | ChatService | `agent-relay/src/orchestrator/ChatService.ts` | 单次对话（`/api/chat` 与 `/api/chat/stream`）：智能路由 / 草拟-审查协作 / 流式 token；从 Orchestrator 抽出以瘦身 |
| 编排 | RunStateStore | `agent-relay/src/orchestrator/RunStateStore.ts` | 预算耗尽续跑：`run_states` 表；`location` 含 searchPlan/visitedFiles/candidateFiles 与 ProjectIndex 统计 |
| 编排 | taskGraph | `agent-relay/src/agent/taskGraph.ts` | 依赖 DAG 校验、`sortSubtasksByPriority` 拓扑+优先级排序 |
| 编排 | TaskRunner | `agent-relay/src/agent/TaskRunner.ts` | 任务**状态机**：pending→running→completed/blocked/failed/skipped；`resume(retry/skip/confirm)`；状态迁移写入 `task_status_change` trace |
| 策略 | policy | `agent-relay/src/policy/` | Workspace / Shell / Network / Permission / **PermissionGuard** / **ToolRiskAssessment**；`StructuredToolRisk` 统一高风险工具输出；`PermissionPolicy` 定义 project→mode→role→task→user 权限覆盖顺序；`PermissionGuard` 在 Agent 工具调用前输出 `allow` / `needsConfirmation` / `deny`，并生成结构化 `confirmationRequest`；删除/清空、提交/推送、远程脚本、系统环境、全局依赖、密钥读写等高风险行为在自动策略下仍强制确认 |
| 编排 | AgentLoop | `agent-relay/src/agent/AgentLoop.ts` | 自主对话循环（M1）：ReAct JSON + 工具迭代；**默认 chat 经 `createAgentChatFn` → SmartModelRouter**（显式 `clientName` 仍走 ModelRouter）；trace 含 `agent_decision` / `agent_model_turn` / `run_usage_summary`；可选接入 `AgentTimelineService` 发布公开执行步骤；工作流执行元信息聚合已下沉到 `workflowExecutionMeta` |
| 编排 | workflowExecutionMeta | `agent-relay/src/agent/workflowExecutionMeta.ts` | 聚合 `executionMeta` 的工作流结果：`workflowDiffs` / `workflowVerifications` / `workflowCorrections` 与 `location`，用于降低 `AgentLoop` 的编排耦合 |
| 编排 | workflowWriteOrchestrator | `agent-relay/src/agent/workflowWriteOrchestrator.ts` | 封装首次写入门控与 phase 注入装配：统一处理 write gate 阻塞、`workflowWritePhases` / `workflowDebugFixes` 记录与待注入上下文 |
| 编排 | workflowFollowupContexts | `agent-relay/src/agent/workflowFollowupContexts.ts` | 统一构建工具执行后的 workflow 附加上下文：blocked/write/execution/verification/correction，并集中返回待清空的 pendingWritePhaseContext |
| 编排 | Agent Activity Timeline | `agent-relay/src/agent/timeline/` | 公开执行过程摘要（非 CoT）：`AgentTimelineService` + `ActivityRunStore`（`.agent/runs/{id}/`）+ `AgentEventBus`；`POST /api/agent/stream` 推送 `activity_event`；`GET /api/agent/runs/:id` / `events` 快照与 SSE 重放 |
| 编排 | RunPolicyManager | `agent-relay/src/agent/RunPolicyManager.ts` | 消费 `IntentRouter` 的内部意图/工作流结果，解析运行模式、分项预算默认值、用户侧 `permissionPolicy` 与 systemHint；工具权限由 `permissionPolicy` 推导，`mode` 负责工作流/预算 |
| 编排 | BudgetManager | `agent-relay/src/agent/BudgetManager.ts` | 分项预算用量统计、耗尽检测、建议预算、PlanWorkflow 配额 |
| 编排 | Finalizer | `agent-relay/src/agent/Finalizer.ts` | 预算耗尽部分收尾；`taskComplexity` 估算 `suggestedToolCalls`；`completedSteps`/`missingSteps` |
| 编排 | AgentToolStep | `agent-relay/src/agent/toolStep.ts` | Agent 工具步骤；含 `resultLayers`（raw/model/user 三层） |
| 工具/通用 | ToolResultLayers | `agent-relay/src/util/toolResultLayers.ts` | 工具结果分层：raw 审计、modelVisible 回灌、userDisplay 展示；纯助手置于 util/，供 agent/ 与 context/ 共享，避免 context→agent 反向依赖 |
| 编排 | RunPolicy | `agent-relay/src/agent/RunPolicy.ts` | Agent 运行模式、用户侧 `permissionPolicy` 策略、默认预算、工具权限边界与 `executionMeta` 停止原因 |
| 编排 | RunPolicyTypes | `agent-relay/src/agent/RunPolicyTypes.ts` | RunPolicy / 预算 / executionMeta 纯类型与解析函数；`executionMeta` 暴露 `modeSource`、`intent`、`workflowType`、`permissionPolicy`，供 Agent/Orchestrator 共享 |
| 编排 | IntentTypes | `agent-relay/src/agent/IntentTypes.ts` | 自动工作流内部意图与工作流类型：`answer` / `plan` / `edit` / `verify` 等纯类型 |
| 编排 | IntentRouter | `agent-relay/src/agent/IntentRouter.ts` | 统一入口意图识别：先推断 `intent`，再兼容映射到现有 `AgentRunMode`，并联动 `WorkflowRouter` / `WorkflowPlanner` |
| 编排 | intentPatterns | `agent-relay/src/agent/intentPatterns.ts` | IntentRouter 与 WorkflowPlanner 共用的意图正则、关键词兜底、意图→运行模式映射与项目范围检测（单一词表来源） |
| 编排 | WorkflowRouter | `agent-relay/src/agent/WorkflowRouter.ts` | P2 工作流路由骨架：从 `intent` 映射到 `workflowType`、执行器标识、只读/副作用类别；`answer` / `summarize` / `search` 工作流强制工具层只读 |
| 编排 | WorkflowPlanner | `agent-relay/src/agent/WorkflowPlanner.ts` | 按模式/目标/intent 选择预扫描工作流（`plan_prescan` / `edit_locate` / `generate_file_locate` / `debug_locate` / `implement_locate`） |
| 编排 | WorkflowExecutor | `agent-relay/src/agent/WorkflowExecutor.ts` | 预模型确定性工作流统一调度入口；当前挂接 `PlanWorkflow`、`DebugAnalysisWorkflow`、`EditProposalWorkflow` 与 `RunVerifyWorkflow`，向 `AgentLoop` 返回新增步骤、模型上下文和工作流审计记录 |
| 编排 | DebugAnalysisWorkflow | `agent-relay/src/agent/DebugAnalysisWorkflow.ts` | debugWorkflow 的只读诊断分析阶段；在定位上下文后注入错误摘要、疑似文件、根因假设、最小修复、验证计划与风险回滚约束，并生成 `executionMeta.workflowDebugAnalyses` 审计记录 |
| 编排 | PlanWorkflow | `agent-relay/src/agent/PlanWorkflow.ts` | 内部确定性只读预扫描执行器，不注册为 `ToolRegistry` 可调用工具；执行链为 `project_scan` → `locate_relevant_files` → `context_pack`（或子集），支持 edit/generate-file 预定位 |
| 编排 | EditProposalWorkflow | `agent-relay/src/agent/EditProposalWorkflow.ts` | edit/generate-file 的写入前方案阶段；不执行工具，只注入 `targetFiles` / `changeSummary` / `permissionCheck` / `diffPlan` / `verificationPlan` 约束，并生成 `executionMeta.workflowProposals` 可审计契约；`permissionChecks` 复用 `PermissionGuard` 预检 `apply_patch` / `write_file` |
| 编排 | EditExecutionWorkflow | `agent-relay/src/agent/EditExecutionWorkflow.ts` | edit/generate-file 的写入后执行阶段上下文；不执行工具，只在 `write_file` / `apply_patch` 成功后把真实 diff/changeId/path 注入下一轮模型，要求最小验证或最终总结 |
| 编排 | EditAutoVerificationWorkflow | `agent-relay/src/agent/EditAutoVerificationWorkflow.ts` | edit/generate-file 的写入后自动验证规划；不直接执行工具，只在写工具返回 path 时生成只读 `read_file` 动作，由 `AgentLoop` 走既有权限、预算与工具链路执行 |
| 编排 | EditVerificationWorkflow | `agent-relay/src/agent/EditVerificationWorkflow.ts` | edit/generate-file/debug 的写入后验证阶段上下文；观察写入后的 read/diff/shell 等验证工具结果，注入 verification phase，并生成 `executionMeta.workflowVerifications` 审计记录 |
| 编排 | WorkflowCorrectionWorkflow | `agent-relay/src/agent/WorkflowCorrectionWorkflow.ts` | 验证失败后的 correction/termination phase；按路径统计 attempt，默认达到 2 次失败即进入 termination，生成 `executionMeta.workflowCorrections` |

| 编排 | WorkflowWriteGate | `agent-relay/src/agent/WorkflowWriteGate.ts` | edit/generate-file/debug/refactor 写入门控；基于 `WorkflowStateCenter` 阻塞缺少计划/只读上下文的首次写入、未验证前的连续写入以及修正上限后的写入 |

## 三、关键调用链路

### 1. 对话请求（`POST /api/chat`，默认 Smart 路由）

未指定 `clientName` 时走 **SmartModelRouter + ModelOrchestrator**；显式 `clientName` 时绕过 Smart，走 **ModelRouter 连通性 fallback**。双轨职责见 [模型路由与协作 · 双轨边界](模型路由与协作.md#双轨路由边界必读)。

```mermaid
sequenceDiagram
    participant U as 网页/CLI
    participant S as HTTP 服务
    participant SR as SmartModelRouter
    participant O as ModelOrchestrator
    participant C as ModelClient
    participant M as route_logs / Metrics

    U->>S: POST /api/chat { message, qualityMode }
    S->>SR: route(RouterInput)
    SR-->>S: RouterDecision
    S->>O: run(decision, prompt)
    alt rule_only
        O-->>S: 规则直答（无模型调用）
    else single_model / 协作
        O->>C: chat(selectedModelId)
        C-->>O: ModelResponse
        opt FallbackManager 升级
            O->>C: 强模型重试
        end
    end
    O-->>S: finalAnswer + executionStrategy
    S->>M: 路由/调用/fallback 日志
    S-->>U: JSON
```

### 1b. 显式 clientName（ModelRouter 客户端层）

```mermaid
sequenceDiagram

    participant S as HTTP 服务
```

### 2. 计划报告 → 编译 → 任务模式

```mermaid
flowchart LR
    A["POST /api/plans/analyze"] --> B["AgentLoop mode=plan<br/>只读扫描"]
    B --> C["UserVisiblePlan<br/>Markdown + TodoList"]
    C --> D["用户确认 Todo"]
    D --> E["POST /api/plans/:userVisiblePlanId/compile"]
    E --> F["PlanCompileWorkflow → InternalTaskPlan draft"]
    F --> G["PlanValidator + PlanStore"]
    G --> H["POST /api/plans/:planId/approve"]
    H --> I["POST /api/plans/:planId/execute"]
    I --> J["TaskExecutionWorkflow"]
    J --> J2["TaskRunner 逐步执行"]
    J2 --> K{"步骤需确认?"}
    K -- 是 --> L["confirm / autoConfirm"]
    K -- 否 --> M["ToolStepExecutor"]
    L --> M
    M --> N{"权限在允许集?"}
    N -- 否 --> O["blocked"]
    N -- 是 --> P["completed / failed"]
```

> **执行边界**：`AgentStepPlan` 只进 trace；`UserVisiblePlan` / `PublicPlanJson` 不可直接执行；`POST /api/task/run` 不再接受 inline `plan` JSON；dry-run 仍兼容 legacy `plan`（自动 ingest）。详见 [计划体系分离](计划体系分离.md) 与 [计划JSON与Markdown分离](计划JSON与Markdown分离.md)。

> 任务真实执行经 `Orchestrator.runTask` → `TaskExecutionWorkflow` → `TaskRunner`（`dependsOn` DAG 波次并行）→ `ToolStepExecutor` → `ToolRegistry`；干跑使用 `DryRunExecutor`。Plan 步骤会同步到 `task_steps` / `task_step_dependencies`，每次执行尝试写入 `task_attempts`，步骤与聚合任务状态变化写入 `task_status_change` trace。请求体 `rollbackOnFailure: true` 时，失败/blocked 会经 `TaskRollback` 逆序调用 `rollback_change` 补偿文件变更。

## 六、设计原则

- **分层单向依赖**：上层依赖下层，模型层不感知编排/服务层，便于替换与单测。
- **接口先行**：`ModelClient`、`StepExecutor` 等以接口隔离实现，新增 Provider / 执行器不动上层。
- **可观测**：所有模型调用经 `MetricsRegistry` 聚合并写入 `TraceLogger`，便于排查与成本核算。
- **安全边界**：按模式划分 `ToolPermission`，高风险步骤需显式确认，敏感任务强制仅本地。
- **零重型依赖**：服务/前端/文档站均不引框架，降低维护与理解成本。
