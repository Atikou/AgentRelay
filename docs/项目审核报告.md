# 项目审核报告（独立定义基线）

> 审核人：AI Agent（Claude）｜审核时间：2026-06-16（Asia/Shanghai）
> 审核对象：AgentRelay（`agent-relay/`，本地优先 Agent 编排后端）
> 触发：用户要求「先修复子 Agent 问题，再判断其余模块是出发点就错了还是中途漂移」。

## 0. 审核方法论（重要）

本报告刻意 **不采用项目自带文档/代码注释里的模块自述** 作为评判标准。
对每个模块，我先从「一个干净的 Agent 编排后端应当如何分层」的第一性原理出发，
**自己给出该模块应当承担的职责定义**，再用实现去对照这条定义，判断它是否偏离、
以及偏离属于「出发点（设计起点）就错」还是「中途（迭代累积）才漂移」。

评级口径：

| 评级 | 含义 |
| --- | --- |
| ✅ 符合 | 实现与我的定义一致，边界清晰 |
| 🟡 部分偏离 | 核心职责对，但夹带了不该有的职责 / 存在跨层耦合 / 有未兑现承诺 |
| 🔴 偏离 | 职责严重越界或与定义冲突，是主要的架构债来源 |
| ⚪ 未兑现 | 模块宣称的目标基本没有落地，近似占位/死代码 |

偏离归因口径：

- **出发点错**：模块从被创建那一刻起，概念建模本身就不符合它应有的角色。
- **中途漂移**：起点正确，但在里程碑迭代（M1–M8、路由 V1–V9、生命周期分期等）中
  不断把相邻职责往里塞，或留下半成品，逐渐失真。

## 1. 子 Agent 专项（本次首要修复，已确认）

### 我的定义

子 Agent 应当是 **主 Agent 的一种「工具」**：主 Agent 在需要把一段可独立完成、
上下文可隔离的子任务外包出去时，像调用任何其他工具一样调用它，传入「要做什么 + 最小上下文 + 权限/模型策略」，
拿回「压缩后的结构化结论」。它 **不应该是一个有固定人格 / 预设身份（role/persona）的角色**——
因为「角色」会带来隐含的能力假设、固定提示词、以及「谁该被召唤」的分类逻辑，
这与「按任务即时派发、用完即弃、上下文干净」的本质相冲突。

### 实现实况（修复后）

- 旧的 `src/subagent/roles.ts`（125 行的角色/预设体系）已在提交 `d29342e` 中**删除**。
- 对外入口只有工具 `dispatch_subagent`（`tools/subagentTool.ts`），入参是 `tasks: DelegatedTask[]`，
  每个任务携带 `goal / instructions / toolPolicy / modelPolicy / outputContract`，**没有任何 role 字段**。
- `dispatch-subagent-tool.test.ts` 含用例「拒绝旧 roles 参数」「子 Agent 内不暴露 dispatch_subagent」，
  从测试层面固化了「工具而非角色 / 不可无限递归」。
- 模型选型走 `createDelegatedTaskChatFn` → `analyzeTaskRoutingSignals`（由任务信号推导能力/质量），
  而不是按角色绑定模型。`subagent-execution-router.test.ts` 用例「按 modelPolicy 选型而非角色」「通用子 Agent 不默认绑定代码能力」佐证。
- 系统提示（`subagent/taskPrompt.ts`）显式声明「你是主 Agent 委派的子任务执行单元，不是独立对话助手……禁止直接回复终端用户、扩大范围、调用 dispatch_subagent」。

### 判定：✅ 符合（修复完成）

子 Agent 现在是纯工具语义。本次审核已确认无残留的角色/人格概念：
代码内剩余的 `role` 字样均为 **消息角色**（system/user/assistant）或 **权限上限**（`roleAllowedPermissions` / `roleSource`），
`config/types.ts` 的 `allowedRoles: [primary/draft/review/final]` 是 **模型协作流水线角色**（草拟/审查），与子 Agent 无关。

### 归因：这是少数「出发点就错」的模块

子 Agent 最初按「角色/预设」建模，是 **概念起点错误**——把「能力分类」错当成「身份分类」。
现已纠正为工具。这也正是用户直觉的来源；但下文会看到，**项目其余大部分问题并非出发点错，而是中途漂移**。

### 仍建议跟进（非角色问题，属诚实性/元数据）

- `dispatch_subagent` 工具元信息为 `permission: "read"`、`hasSideEffect: false`，但当 `toolPolicy.writeAllowed` 且授予 `write` 时，子 Agent 可写盘。
  对上层权限/确认门而言，这是「副作用未如实声明」。建议：要么按运行期参数动态上报副作用，要么在工具描述与确认门里显式提示「子任务可能写盘」。

## 2. 逐模块审核

### 模型层

#### 2.1 `model/` — 🟡 部分偏离（中途漂移）

- **我的定义**：纯供应商适配层。只负责把统一的 `chat()` 翻译成各家协议（Ollama/OpenAI 兼容/Anthropic）、处理流式与可用性探测。**不得包含路由、fallback、任务分类、业务逻辑**。
- **实现实况**：`ModelFactory / OllamaClient / OpenAICompatibleClient / AnthropicClient / ModelCatalog` 完全符合定义；但模块内还有 `ModelRouter.ts`（策略排序 + fallback 链 + 远程脱敏 + 指标）和 `taskType.ts`（按任务类型重排候选）——这是路由职责，`MetricsRegistry` 是观测职责。
- **判定**：核心适配器干净，但 `ModelRouter`/`taskType` 属于路由逻辑下沉到了模型层。
- **归因**：典型中途漂移。在 `model-router/` 诞生之前，先在 `model/` 里建了一套「遗留路由」；新路由建好后 **遗留路由从未退场**，形成「双路由轨道」（见 §3.1）。出发点（适配层）是对的，是迁移做了一半。

#### 2.2 `model-router/` — 🟡 部分偏离（中途漂移）

- **我的定义**：**决策层**。给定任务与上下文，决定「用哪个/哪些模型 + 什么执行策略」，并记录决策。**只决策、不执行模型调用**。
- **实现实况**：决策核心（`RuleRouter → DecisionEngine → ContextAnalyzer`、能力矩阵、可用性预检）扎实且符合定义。但模块还塞进了：
  - 执行胶水 `create-smart-single-model-chat.ts` / `create-subagent-chat.ts` / `create-planner-chat.ts`——它们 **import 了 `agent/AgentLoop` 与 `subagent/`**，是集成/执行层。
  - 持久化 `route-stores.ts`（420 行 SQLite）、离线评测子系统 `eval-set-*`（4 文件）、提示策略构建。
  - 与 `model-orchestrator/` 形成双向（类型/工厂）依赖。
- **判定**：决策核心符合定义，但累积了执行胶水与持久化，「只决策」的边界被冲淡。
- **归因**：中途漂移。各执行路径（Agent/Planner/SubAgent）的 chat 工厂被就近放在路由模块里图方便，导致路由层反向依赖 `agent/`、`subagent/`。

#### 2.3 `model-orchestrator/` — ✅ 符合（轻微耦合）

- **我的定义**：**执行层**。拿到一个 `RouterDecision` 后真正跑策略（单模型直跑，或本地草拟+远程审查协作），返回最终答案。应依赖「决策契约」，不重复实现路由。
- **实现实况**：只含 pipelines（single-model / draft-review / rule-only）+ 提示模板，未掺路由规则。依赖 `model-router` 的 fallback/answer-eval/协作存储。
- **判定**：守住了本分。
- **归因**：无出发点问题。小瑕疵：`rule_only` 在此与各 chat 工厂里重复实现；与 `model-router` 的双向依赖。属正常迭代留下的小重复，非漂移。

### 自治执行层

#### 2.4 `agent/` — 🟡 部分偏离（中途漂移，god-object 化）

- **我的定义**：**自治执行大脑**。给定目标，跑 ReAct 循环（模型→工具→观察→收敛），在预算/权限约束内迭代到答案。应拥有「循环本身及其控制流辅助」，把「计划持久化、权限裁决、模型选型」委派给专门模块。
- **实现实况**：概念正确且处于中心位置，但：
  - `AgentLoop.ts` 1568 行——循环 + 10+ 个 `Workflow*` 阶段 + timeline + 写入门 + trace + 子 Agent 派发守卫 + 续跑状态，全压在一个类里（god-object）。
  - 与 `plan/`（双向：用 `AgentStepPlan`，又被 `plan/` 用 `Planner`/`taskGraph`/`types.Plan`）、`policy/`（双向）、`orchestrator/`（双向，`ApiResult`/`runStateTypes`）存在环状依赖。
  - 两套模式词汇并存：`permissions.ts` 的遗留 `AgentMode = plan|task` vs `RunPolicyTypes` 的 `AgentRunMode = chat|plan|implement|debug|review`。
  - `IntentRouter` 与 `WorkflowPlanner` 各有一份重叠的中英文意图正则。
- **判定**：循环这个核心是对的，但过度中心化 + 与 plan/policy/orchestrator 的边界含糊。
- **归因**：中途漂移。自动工作流 P0–P6 一路往 `AgentLoop` 上叠，没有及时抽出独立的工作流编排层。

#### 2.5 `plan/` — 🟡 部分偏离（中途漂移，模型膨胀）

- **我的定义**：**计划域**。表示、校验、持久化、审批并交付「可执行任务计划」。应拥有计划数据与生命周期，且有 **单一、清晰的可执行表示**。
- **实现实况**：审批/持久化核心合理（`PlanStore`/`PlanService`/`PlanValidator`/审批），但存在 **四种计划形态**：`UserVisiblePlan`（不可执行，给人看）、`InternalTaskPlan`（唯一可执行、需审批）、`AgentStepPlan`（瞬态 trace）、以及遗留的 `agent/types.Plan`，靠 `planConverter.ts` 在 `InternalTaskPlan ↔ 遗留 Plan` 间手工转换。可执行类型仍住在 `agent/`，执行仍走 `agent/TaskRunner`，导致 `plan/` 无法独立成层。
- **判定**：核心对，但计划模型膨胀 + 依赖 `agent/` 才能闭环。
- **归因**：中途漂移。逐步引入新的计划层级（用户可见计划、内部计划），却没有退役遗留 `Plan`，留下转换垫片。

#### 2.6 `tools/` — 🟡 部分偏离（中途漂移 + 元数据不诚实）

- **我的定义**：一组 **离散、沙箱化的能力**。每个工具自包含，**如实声明自己的 permission/副作用**，且 **不得内嵌编排逻辑或项目专属启发式**。
- **实现实况**：绝大多数工具干净（文件/shell/git/diff）。但：
  - `locationTools.ts` 1374 行——把 scan/index 同步/排序/语义/历史/依赖图召回揉成一个迷你子系统，并 **硬编码了仓库专属路径提示**（`src/plan`、`src/agent`、`agent-relay/src`…）和 `projectId: "default"`。
  - `dispatch_subagent` 的副作用声明与实际不符（见 §1）。
  - `tools/` 反向 import 了 `agent/`（`ExplorationProgressTracker`、`permissions`）。
- **判定**：工具抽象对，但 `locationTools` 越界成子系统，且通用工具里塞了项目专属启发式。
- **归因**：中途漂移（定位/检索能力不断加码）；副作用元数据问题则更像「出发点没想清楚」。

### 知识与存储层

#### 2.7 `context/` — 🟡 部分偏离（中途漂移 + 反向依赖）

- **我的定义**：**记忆与检索基座**。持久化会话/记忆、索引项目、为每一轮组装合适的上下文包/提示。应是知识底座，**独立于 Agent 如何格式化工具输出**。
- **实现实况**：能力完整。但 `ContextManager` **import 了 `agent/ToolResultLayers`**（反向依赖，知识层依赖了执行层的格式化逻辑）；`memory.db` 经迁移承载了 plan/routing/eval/run/project 等多域表，使「上下文库」事实上成了全局库；`ApiEmbeddingProvider`、`ModuleDependencyGraph` 等导出未被实际接线；同样硬编码了仓库路径启发式。
- **判定**：底座扎实，漂移在「反向依赖 agent/」与「共享库耦合」。
- **归因**：中途漂移。

#### 2.8 `storage/` — 🟡 部分偏离（中途漂移）

- **我的定义**：**通用、带版本的 SQLite 迁移框架**——与具体业务 schema 无关的管道。
- **实现实况**：`sqliteMigration.ts` 通用、合格。但 `memoryDbMigrations.ts` **import 了 `model-router` 的建表辅助**（`ensureRoutingTables`/`ensureEvalTables`），并拥有 routing/eval/plan/run/project 等业务表 DDL；文件头注释「v1–v7」与实际 `MEMORY_DB_SCHEMA_VERSION = 11` 不符。
- **判定**：框架符合定义；`memory.db` 迁移集越界，把多域业务 schema 集中到了存储层并反向依赖路由模块。
- **归因**：中途漂移。

### 编排与边界层

#### 2.9 `orchestrator/` — 🔴 偏离（主要架构债 sink）

- **我的定义**：**应用门面**。把 API 请求映射到领域工作流，拥有统一 Run 记录，返回传输形态结果。**薄协调，不放业务逻辑**。
- **实现实况**：`Orchestrator.ts` **2001 行**——chat 智能路由内联（~140 行）、plan、task、agent、subagent、scheduler 触发、run CRUD、session/task 绑定全在一个类；`prepareAgentRun` 与 `AgentLoop` 构造重复；`/api/plan` 与 `/api/plans/draft` 双入口。
- **判定**：门面思路对，但吸收了大量本该在 agent/plan/chat 服务里的业务逻辑，成为全项目最大的 god-object 与漂移汇聚点。
- **归因**：中途漂移最严重的一处——「为了避免 handler 膨胀」把逻辑搬进了 orchestrator，却没建立中间服务层。

#### 2.10 `policy/` — 🟡 部分偏离（出发点的层序倒置）

- **我的定义**：**权限/风险裁决权威**。权限求交、风险分级、确认门、沙箱。应是 **更底层、自包含** 的一层，供循环查询。
- **实现实况**：功能活跃且中心化。但它 **依赖 `agent/`**（`permissions`、`IntentTypes`、`RunPolicyTypes`），且把 `agent/permissions.js` 的权限常量再导出——即「权限词汇的规范定义住在 `agent/`，policy 反过来消费」。`WorkspacePolicy` 是对 `tools/pathSafe` 的薄转发，疑似未被使用。
- **判定**：职责对，但层序倒置——它不是干净的下层。
- **归因**：偏出发点。权限词汇最初诞生在 `agent/`，`policy/` 是后来叠在其上的，规范定义没有下沉到 policy 或更底层。

#### 2.11 `server/` — 🟡 部分偏离（边界基本守住，局部渗漏）

- **我的定义**：**HTTP 传输边界**。解析、校验、路由到服务、序列化。handler 必须薄，**不放业务逻辑**。
- **实现实况**：多数 handler 很薄；但 `tools.handlers` 内联了写/补丁/shell 的 preview 构建、`plan.handlers` 直接实例化工作流、`runs.handlers` 在 handler 里拼 runReport、`routing.handlers` 每请求 new 一个 `RuntimeStatsCollector`；`createHttpServer.ts` 是 544 行扁平 if/else 路由；`handleContextSessionDelete` 是已导出但未接线的死代码；plan 草拟双路由。
- **判定**：边界大体守住，少数 handler 渗漏业务逻辑、缺统一服务层。
- **归因**：中途漂移（路由表与 handler 随特性累积膨胀）。

### 基础设施与支撑层

#### 2.12 `core/` — 🟡 部分偏离（未被充分采用）

- **我的定义**：**最底层共享内核**。跨切面原语（Run 身份、关联上下文），所有层都可依赖，零向上依赖。
- **实现实况**：含 `RunKind/RunStatus/RunRecord` + correlation，零向上依赖（结构干净）。但 **只有 `orchestrator/` 用它**；Agent 续跑状态类型却住在 `orchestrator/runStateTypes.ts`；`newRequestId()` 未被使用。
- **判定**：干净但被架空，没真正充当共享契约。
- **归因**：中途漂移（其它模块各自造类型，没回填到 core）。

#### 2.13 `background/` — ✅ 符合

- **我的定义**：管理带外/异步工作（spawned 进程）+ 一个 Agent 在安全点消费的通知通道。自包含基础设施。
- **实现实况**：`BackgroundTaskManager` + `NotificationQueue` 与定义吻合，依赖 `tools/policy/trace` 合理。
- **判定**：守在本分。小瑕疵：后台任务仅存内存（无持久化）、`OUTPUT_MATCH_PRESETS` 模块外未引用。

#### 2.14 `scheduler/` — ✅ 符合

- **我的定义**：时间/事件触发注册表，触发后只 **发通知/请求建 Run**，**不得自己执行工具**（与执行解耦）。
- **实现实况**：once/interval/cron/event + journal，明确「不直接执行工具」（`Scheduler.ts` 注释佐证）。
- **判定**：设计正确且自律。小瑕疵：`Scheduler.ts` 515 行；lifecycle 策略里声明的 journal 保留期无人实现。

#### 2.15 `lifecycle/` — 🟡 部分偏离（承诺先行、实现滞后）

- **我的定义**：**数据治理**。盘点、保留期、安全清理、隐私清除。应在不纠缠业务逻辑的前提下治理存储，且 **实现要对得上自己声明的策略**。
- **实现实况**：preview/apply/purge 主链路扎实（只清 `canDelete && risk=low`）。但 `types.ts`/`policy.ts` 声明了一大批 **未实现** 的策略：gzip/zstd trace 压缩、`delete_db_rows`/`vacuum_db` 动作、`traceRaw*`/`toolArgs`/`routeDetails` 等行级保留、quotas、scheduler journal 保留；`loadPreviewsFromDisk()` 是空桩；`CleanupJournal.listRecent()` 死导出；通知压缩逻辑在 planner/executor 里重复。
- **判定**：最大问题是「策略远超实现」——对外承诺了它并不提供的治理能力。
- **归因**：中途漂移（分期路线图把策略 schema 先铺开，落地没跟上）。

#### 2.16 `trace/` — 🟡 部分偏离（中途漂移 + 半成品索引）

- **我的定义**：**追加式可观测性**。结构化事件日志、分段/轮转、查询/回放/报告。基础设施，理想上只依赖 `util`。
- **实现实况**：日志/分段/轮转/查询合格。但 `runReport.ts`（420 行）**import 了 `model-router` 路由存储** 做报告富化（跨域）；模块内 `traceCatalog ↔ traceQuery ↔ traceReader` 互相引用（环）；trace 索引的 `byte_offset/byte_length` 列从不写入（半成品）；`activeSegmentRel()` 死导出。
- **判定**：核心日志符合定义，报告富化与未完成索引是漂移。
- **归因**：中途漂移。

#### 2.17 `config/` — ✅ 符合

- **我的定义**：类型化、带校验的配置加载——干净叶子模块。
- **实现实况**：干净叶子，无向上依赖。schema 偏宽（含 scheduler/security/router profile），属可接受的「集中配置」。
- **判定**：符合定义。

#### 2.18 `cli/` — 🟡 部分偏离

- **我的定义**：**薄命令行入口**，复用 app 的组装，而非重新实现。
- **实现实况**：`storage.ts` **重新实现了 DI 接线**（不复用 `createAppContext`），且 `getActiveRunIds: () => []` 意味着 CLI 清理不会保护「活跃」run；存在 `main.ts storage` 与直跑 `storage.ts` 双入口。
- **判定**：能用，但重复了 bootstrap 而非复用。
- **归因**：中途漂移（CLI 工具就近自给）。

#### 2.19 `util/` — ✅ 基本符合

- **我的定义**：**零依赖、广泛共享的纯工具**。无领域知识、无向上依赖。
- **实现实况**：层序干净，无向上依赖。小瑕疵：`patch.ts` 仅被测试使用（src 内死代码）、`hasSensitiveValue` 未被引用、`util/costBudget` 与 `model-router/cost-budget-manager` 命名易混。
- **判定**：符合定义，少量死代码。

#### 2.20 `types/` — ⚪ 未兑现

- **我的定义**：若存在，应是 **真正的共享类型聚合点**。
- **实现实况**：15 行占位，仅 re-export `model/types`，**全仓库无人 import**；注释里「随里程碑聚合类型」的目标从未落地。各域各自维护类型 barrel。
- **判定**：近似死占位，未兑现自身目标。
- **建议**：要么真正在此聚合并全仓采用，要么删除以消除误导。

## 3. 跨模块系统性结论

### 3.1 双路由轨道（最大的「迁移做了一半」）

项目同时跑两套路由：

| 路径 | 路由 | 执行 |
| --- | --- | --- |
| `POST /api/chat`（默认） | Smart（`model-router`） | `ModelOrchestrator`（可协作） |
| `POST /api/chat/stream` | 遗留 `ModelRouter`（`model/`） | 直连 client |
| AgentLoop / SubAgent / Planner | Smart（强制单模型） | `createModelChatFn` 直连 |
| 显式 `clientName` | 遗留 `ModelRouter` | 直连 client |

遗留 `ModelRouter` 住在本应是纯适配层的 `model/`，新 Smart 路由建好后未退役，
导致「同一件事两套实现 + 流式与非流式架构分叉」。这是中途漂移的标志性案例。

### 3.2 god-object 群（按累积膨胀形成）

`Orchestrator.ts`（2001）、`AgentLoop.ts`（1568）、`locationTools.ts`（1374）、`createAppContext.ts`（706）、
`stores.ts`（660）、`fileTools.ts`（608）、`Scheduler.ts`（515）。
它们都是「相邻职责不断往里塞」的产物，而非一开始就设计成巨石。

### 3.3 层序倒置 / 环状依赖（部分出发点、部分漂移）

- `policy/ → agent/`（权限规范住在 agent，policy 反向消费）—— **偏出发点**。
- `context/ → agent/ToolResultLayers`、`storage/ → model-router`、`trace/runReport → model-router` —— **中途漂移**。
- `agent/ ↔ orchestrator/ ↔ plan/` 三角环 —— **中途漂移**。

### 3.4 承诺先行、实现滞后（spec-ahead-of-implementation）

`lifecycle/` 保留期与压缩、`trace/` 字节索引、`types/` 聚合点——都属于「schema/接口先铺、落地没跟上」。
这不是出发点错，而是路线图驱动开发留下的半成品。

### 3.5 总体回答用户的问题

> 「是出发点就错了，还是中途出现了问题？」

- **出发点就错的**：仅 **子 Agent（角色 vs 工具）**（已修复）；以及 `policy/` 的权限层序倒置（轻度）。
- **绝大多数是中途漂移**：双路由轨道、god-object 群、跨层反向依赖、承诺先行未落地。
  根因是 **里程碑/版本驱动的特性累积**（M1–M8、路由 V1–V9、生命周期分期）在「就近图方便」时
  反复越过模块边界，且 **旧实现/旧 schema 没有及时退役**。

模块的「分层意图」本身基本是正确的——问题出在 **执行纪律**（边界维护、迁移收尾、抽取中间层）。
