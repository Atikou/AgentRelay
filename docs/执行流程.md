# 执行流程

本文描述用户请求在系统中的端到端路径。配合 [架构设计](架构设计.md) 阅读。

## 1. 统一 Agent 请求（主路径）

```mermaid
sequenceDiagram
  participant U as 用户/测试台
  participant API as POST /api/agent
  participant O as Orchestrator
  participant RP as RunPolicyManager
  participant EIR as EntryIntentRouter
  participant AL as AgentLoop
  participant MR as SmartModelRouter
  participant T as ToolRegistry

  U->>API: message + sessionId + permissionPolicy
  API->>O: runAgent / runAgentStream
  O->>O: tryPlanHandoffFollowUpResume（短句续执行）
  O->>O: findBlockingAgentPause（409 若待批准）
  O->>RP: resolve
  RP->>EIR: 会话延续 / AI / legacy
  O->>AL: run(policy)
  AL->>AL: WorkflowExecutor 预扫描
  loop ReAct
    AL->>MR: chat
    MR-->>AL: JSON action
    AL->>AL: PermissionGuard
    AL->>T: tool（若允许）
    T-->>AL: 结果
  end
  AL-->>O: answer + executionMeta
  O-->>U: JSON / SSE done
```

### prepareAgentRun 步骤

1. 校验 `message`、`mode`、`permissionPolicy`、`taskType`
2. `RunPolicyManager.resolve` → `EntryIntentRouter`
3. `ensureSession`（持久化会话）
4. `findBlockingAgentPause`：有待处理 planHandoff / permission / paused snapshot → **409**
5. `resolveOrCreateTask`：会话内活动任务
6. 创建 `Run`（`kind: agent`）
7. 构造 `AgentLoop` 并执行

## 2. 入口意图决策

```mermaid
flowchart TD
  M[用户消息] --> E{forceMode?}
  E -->|是| X[explicit_mode]
  E -->|否| S[SessionTaskManager.getContext]
  S --> C{ContinuationDetector}
  C -->|continuation| I[继承上轮 intent/workflow]
  C -->|new_task| N[markInactive]
  C -->|uncertain| A[AIIntentClassifier]
  N --> A
  A -->|null/低置信| L[LegacyIntentFallback]
  A -->|高置信| D[IntentDecision]
  L --> H{活跃任务 + answer?}
  H -->|是| I
  H -->|否| D
  I --> W[WorkflowRouter + RunPolicy]
  D --> W
```

### 延续典型场景

| 场景 | 行为 |
| --- | --- |
| 上轮 edit，粘贴 `#2 read_file [error]` | `session_continuation` → 继续 edit |
| 上轮失败，补充日志 | 延续原 intent |
| 短句「继续改」 | 延续（plan 除外，走 planHandoff） |
| 「换个问题」 | `new_task`，不继承 |
| 活跃任务 + 模糊输入 | 覆盖 legacy `answer`，不掉 chat |

`executionMeta.intentDecisionSource`：`session_continuation` | `ai_classifier` | `legacy_fallback` | `explicit_mode`。

## 3. AgentLoop ReAct 循环

```mermaid
flowchart TD
  START[用户消息 + system 工具清单] --> PRE{WorkflowExecutor}
  PRE -->|预扫描步骤| MODEL[模型 turn]
  PRE -->|无| MODEL
  MODEL --> PARSE{解析 JSON}
  PARSE -->|失败| ERR[system 纠偏] --> MODEL
  PARSE -->|final| END[收尾 + executionMeta]
  PARSE -->|tool| GUARD[PermissionGuard]
  GUARD -->|deny/需确认| PAUSE[JIT 暂停或 blocked 回灌]
  PAUSE --> MODEL
  GUARD -->|allow| TOOL[ToolRegistry.run]
  TOOL --> MODEL
  MODEL -->|预算耗尽| PARTIAL[Finalizer partial]
```

### 消息角色

| role | 来源 |
| --- | --- |
| `user` | 仅真实用户输入 |
| `assistant` | 成功解析的 ReAct 动作 |
| `tool` | 工具结果（发送前经 messageBoundary 渲染） |
| `system` | 工作流切换、handoff、纠偏、通知 |

计划 handoff 后模型先输出普通文本 → 仅内存纠偏 + `parse_error` trace，**不**写入持久化 assistant 历史。

## 4. 计划 → 执行（planHandoff）

```mermaid
flowchart LR
  P[plan 意图 final] --> H[PlanHandoffStore.create]
  H --> UI[测试台：按计划执行]
  UI -->|批准| R[resume-plan-handoff]
  R --> AL[AgentLoop resume implement]
  AL --> W[写文件时 JIT permissionRequest]
```

- **planHandoff**：「方案是否执行」
- **permissionRequest**：「这个工具调用是否授权」
- 禁止短句从 plan 直接跃迁 execute（须走 handoff 批准）

## 5. JIT 工具权限

```mermaid
sequenceDiagram
  participant AL as AgentLoop
  participant PG as PermissionGuard
  participant PS as PausedRunStore
  participant PR as PermissionRequestStore
  participant U as 用户

  AL->>PG: 工具调用前评估
  PG-->>AL: needsConfirmation
  AL->>PS: 保存消息链快照
  AL->>PR: permissionRequest
  AL-->>U: stopReason awaiting_permission
  U->>PR: 批准
  U->>AL: resume-permission
  AL->>AL: resumePendingAction 执行被拦工具
  AL->>AL: 继续循环
```

`SessionPermissionGrants`：「本次会话都允许」scoped grant。

## 6. 其他续跑路径

| 路径 | 触发 | API |
| --- | --- | --- |
| 预算耗尽 | `RunStateStore` 有 resumable 状态 | `POST /api/agent/resume` |
| 工具权限 | JIT 暂停 | `POST /api/runs/:id/resume-permission` |
| 计划批准 | planHandoff pending | `POST /api/runs/:id/resume-plan-handoff` |
| 已审批计划 | InternalTaskPlan approved | `POST /api/tasks/:id/resume` |

## 7. 流式响应（SSE）

`POST /api/agent/stream` 事件类型：

| 事件 | 内容 |
| --- | --- |
| `activity_event` | 公开执行过程（Timeline） |
| `model_turn` | 思考摘要（非 CoT 落盘） |
| `token` | 可选流式文本 |
| `done` | 完整结果 + `executionMeta` |

取消：`POST /api/runs/cancel`。

## 8. 聊天与计划 API（旁路）

| API | 用途 |
| --- | --- |
| `POST /api/chat` | 单次对话 + Smart 路由 |
| `POST /api/plans/analyze` | 生成 UserVisiblePlan |
| `POST /api/plans/:id/compile` | 编译 InternalTaskPlan |
| `POST /api/plans/:id/execute` | 已审批计划执行 |

这些路径不经 `EntryIntentRouter`，但有各自的 `Planner` / `TaskRunner` 策略。

## 9. UI 状态展示

测试台主标签读取顺序：

1. `executionMeta.userFacingLabel`（优先）
2. `workflowTaskState` / `workflowType` 映射
3. 详情行可展开：`intent`、`executionStage`、`mode（调试）`

计划交接面板 ↔ `planHandoff`；工具权限面板 ↔ `permissionRequest`。**两者不可混用同一弹窗语义。**

## 10. 追踪与复盘

| 产物 | 路径/API |
| --- | --- |
| Trace JSONL | `data/traces/trace.jsonl` |
