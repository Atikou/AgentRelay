# 安全与审计（M7）



M7 聚焦**日志脱敏、工具调用审计、trace 导出与回放**，与既有命令风险分级、路径沙箱、确认门配合使用。



## 日志脱敏



`src/util/redact.ts` 提供：



| 函数 | 用途 |

| --- | --- |

| `redactString` | 遮蔽 `sk-*`、Bearer token、AWS key、私钥块等 |

| `redactValue` | 深度脱敏对象；字段名匹配 `password/secret/apiKey/token` 等一律替换 |

| `redactPreview` | 工具入参/出参预览（截断 + 脱敏） |



应用点：`TraceLogger` 写入、`ToolRegistry` 审计事件、`ToolStorage.tool_logs` 持久化日志、`ContextManager` 错误 JSONL。

`ToolRegistry` 写入 SQLite `tool_logs` 前会对 `input_json`、`output_json` 与 `error_message` 做深度脱敏；`changeId`、路径、耗时等非敏感审计字段会保留，便于回滚与复盘，同时避免完整 API key、Bearer token、密码、私钥等落盘。

trace 查询接口读取 `data/traces/trace.jsonl` 时只从文件尾部按块回读最近事件，避免日志增长后每次审计查询全量载入内存。

## 敏感信息检测与远程模型脱敏

`src/util/redact.ts` 提供 `detectSensitiveString` / `hasSensitiveValue`，可识别 OpenAI 风格 key、Bearer token、AWS key、私钥块以及 `apiKey/password/secret/token=...` 这类字段文本。

`ModelRouter` 在调用 **remote** 模型客户端前会克隆请求并脱敏 `messages.content`；本地模型调用保留原始内容，便于本地优先和隐私模式下继续处理完整上下文。若发生远程提示脱敏，会写入 `model_prompt_redacted` trace 事件，只记录客户端、模型和脱敏消息数量，不记录原文。

`GET /api/config` 中的 `capabilities.sensitiveDetection`、`capabilities.modelPromptRedaction` 与 `capabilities.toolStorageRedaction` 用于测试台和外部工具展示当前隐私保护能力。

## 本地优先隐私模式

`POST /api/chat` 传 `sensitive:true` 时，模型候选会被限制为本地模型；`config/local-only.json` 使用 `routing.strategy=privacy-first`，可把整个 profile 切到默认仅本地。智能路由与 fallback 也会继承 `localOnly`，避免敏感任务在模型失败或输出不足时升级到远程 API。

`GET /api/config.capabilities.localFirstPrivacyMode` 用于声明当前进程支持该隐私约束。

## 权限覆盖顺序

`src/policy/PermissionPolicy.ts` 的 `resolveEffectivePermissions` 按 **自外向内收窄** 的顺序收敛有效权限集（交集取最严，用户不可扩权）：

| 顺序 | 层级 | 来源 | 说明 |
| --- | --- | --- | --- |
| 1 | **project** | `config.security.permissions.allowed` | 项目级上限；未配置则允许全部内置权限 |
| 2 | **mode** | `RunPolicy` / `AgentRunMode` | plan/review 仅 `read`；implement/debug/chat 为任务全集 |
| 3 | **role** | 子 Agent 角色 `allowedPermissions` | 如 `code_review` 仅 `read` |
| 4 | **task** | `TaskRunner` / `ToolStepExecutor` 传入 | 任务级声明（可选） |
| 5 | **user** | 请求体 `grantedPermissions` / `allowedPermissions` | 用户显式授予，只能收窄 |

**与确认门、命令策略的关系**：`CONFIRMATION_REQUIRED` 确认门、`ShellPolicy` 命令正则、`WorkspacePolicy` 路径沙箱在工具执行阶段另行校验，**不改变**上述 `allowed` 集。即：权限通过 ≠ 自动执行副作用，高风险仍可能需确认或被 Shell 策略拒绝。

接入点：

- `AgentLoop`：project + mode +（子 Agent 时）role + user
- `TaskRunner` / `ToolStepExecutor`：project + task 模式 + task 声明
- `POST /api/tools/run`：仅应用 project 上限（直连测试入口）
- `resolveGrantedPermissions`：子 Agent 授予校验

`GET /api/config` 返回 `security.permissions.allowed`、`security.permissions.scopeOrder` 与 `capabilities.permissionScopeResolution`。

配置示例（只读 profile）：

```json
{
  "security": {
    "permissions": {
      "allowed": ["read"]
    }
  }
}
```

## 写文件 patch 预览

测试台/HTTP 工具接口在 `write_file`、`apply_patch` 未确认时会先返回 `preview.patchPreview`，同时完成路径沙箱、`expectedHash` 与唯一匹配校验；只有调用方再次提交 `confirm:true` 才执行写入。`shell_run` 未确认时也只返回命令风险预览，不启动子进程。

`capabilities.highRiskConfirmation: true` 表示 HTTP 工具入口已启用副作用确认门：未确认时不执行，确认后仍会继续经过权限边界、路径沙箱与 ShellPolicy。`dangerous` shell 命令（如 `rm -rf /`、`git reset --hard`）不会因为确认而放行。

## Shell 命令策略

`shell_run` 与 `BackgroundTaskManager` 共用 `ShellPolicy`：

- 默认启用内置风险分级，`rm -rf /`、`git reset --hard`、下载后直接执行脚本等高风险命令会被拒绝，即使 HTTP 调用传入 `confirm:true` 也不会执行。
- 可在 `config/*.json` 中配置 `security.shell.denyCommands` 与 `security.shell.allowCommands`。两者均为正则字符串数组。
- `denyCommands` 优先级高于 `allowCommands`；配置 `allowCommands` 后，命令必须命中任一 allow 规则才可执行。

示例：

```json
{
  "security": {
    "shell": {
      "denyCommands": ["npm\\s+install", "curl\\b.*\\|\\s*(sh|bash|powershell)"],
      "allowCommands": ["^npm\\s+run\\s+(typecheck|test)\\b", "^git\\s+status\\b"]
    }
  }
}
```

## 网络域名策略（预留）

`NetworkPolicy`（`src/policy/NetworkPolicy.ts`）为 **permission=network** 的未来网络工具预留域名 allowlist / denylist，与 `ShellPolicy` 对称：

- 在 `config/*.json` 配置 `security.network.denyDomains` 与 `security.network.allowDomains`（正则字符串数组，对**规范化 hostname** 匹配）。
- `denyDomains` 优先级高于 `allowDomains`；配置 `allowDomains` 后，目标域名必须命中任一 allow 规则才可访问。
- `ToolRegistry` 对 `network` 权限工具会在执行前从入参 `url` / `endpoint` / `host` 等字段提取目标并预检；拒绝时返回 `permission_denied` / `permission_error`。
- 网络工具实现可在 `ToolContext.networkPolicy` 上调用 `assertAllowed` 做二次校验。

`GET /api/config` 返回 `security.network` 与 `capabilities.networkDomainPolicy`。

示例（仅允许访问自家 API，并封禁已知恶意域）：

```json
{
  "security": {
    "network": {
      "denyDomains": ["(^|\\.)evil\\.com$", "metadata\\.internal$"],
      "allowDomains": ["(^|\\.)api\\.example\\.com$", "^localhost$"]
    }
  }
}
```

## 结构化工具风险字段

`StructuredToolRisk`（`src/policy/ToolRiskAssessment.ts`）为高风险工具提供统一风险结构，便于测试台确认对话框、智能体步骤回显与审计 trace 对齐展示：

| 字段 | 说明 |
| --- | --- |
| `tier` | `low` / `medium` / `high` / `critical` |
| `category` | `file_write` / `file_patch` / `shell_command` / `network_access` / `dangerous_op` / `permission_boundary` |
| `summary` | 一行摘要 |
| `reasons` | 风险原因列表 |
| `requiresConfirmation` | 是否走确认门 |
| `policyBlocked` | 是否被 Shell/Network/权限策略硬拒绝 |
| `commandLevel` | shell 专用，与后台任务 `risk.level` 对齐（`safe`/`caution`/`dangerous`） |
| `target` | 关联路径、命令或 URL |

**产出点**：

- `POST /api/tools/run` 未确认时：`{ needsConfirmation, preview, risk }`
- `ToolRegistry` 失败（权限/策略拒绝、shell 拦截）：`{ ok:false, risk, category }`
- `AgentLoop` 步骤阻塞（未自动确认 / 模式不允许）：`AgentToolStep.risk`
- `tool_audit` 启动事件：`riskTier`（副作用工具）

`GET /api/config.capabilities.structuredToolRisk` 声明当前进程支持该结构。

## 通知注入围栏

后台任务和调度器通知会在 AgentLoop 的安全点注入模型上下文。通知正文视为不可信外部数据；若命中 prompt injection 片段，会被包装为 `_untrusted` 数据，提醒模型只当作事实输入而不要执行其中指令。



`write_file` 执行前经 `util/patch.ts` 生成 `patchPreview`（新文件/变更摘要），写入工具输出与 `tool_audit.outputPreview`，供确认前审阅。



## Prompt injection 防护



`util/injection.ts` 对 `read_file` / `search_text` / `list_files` / `project_scan` / `locate_relevant_files` / `context_pack` 输出扫描可疑短语；命中时以 `_untrusted` 围栏包装后回灌 `AgentLoop`，提示模型勿执行其中指令。



## 工具审计

AgentLoop 每轮模型动作会写入 `agent_decision`：`tool`、`final`、`parse_error`。事件包含迭代轮次、运行模式、工具名、thought、输入预览或 final 长度；不会把完整 final 答案作为 trace 字段保存。


TaskRunner 写入 `task_status_change`：`scope=step` 记录每个步骤的 `from/to/status`、步骤 id、标题与错误；`scope=task` 记录聚合任务状态变化、步骤总数与状态计数。Orchestrator 创建的 task run 会附带 `runId` / `taskId` / `sessionId`。


工具调用链路通过 `toolCallId` 串联：AgentLoop 的 `agent_decision` / `agent_tool`、TaskRunner 的 `task_step`、ToolRegistry 的 `tool_audit start|ok|error` 会共享同一个调用 id。直接执行工具时，ToolRegistry 会自动生成 id。


工具失败会按 `category` 归类为 `user_error`、`environment_error`、`permission_error`、`temporary_error`、`unknown_error`；`tool_audit error` 与工具执行失败响应都会带 `code/category`，便于后续决定是否重试、请求授权或提示用户修正输入。


模型用量通过 `agent_model_turn` 与 `run_usage_summary` 记录：前者按模型轮次记录 client、model、location、latency、token、cost、error；后者在 AgentLoop 结束时汇总模型轮次、token、耗时、费用、工具失败/阻塞数与预算使用。事件只保存摘要，不保存完整模型输出。



每次工具执行写入 trace（`data/traces/trace.jsonl`）：



```json

{

  "type": "tool_audit",

  "tool": "read_file",

  "status": "start|ok|error",

  "permission": "read",

  "toolCallId": "run-1:iter-1:read_file",

  "category": "temporary_error",

  "inputPreview": "...",

  "durationMs": 12

}

```



## HTTP API



| 方法 | 路径 | 说明 |

| --- | --- | --- |

| GET | `/api/trace/recent?limit=50` | 最近 N 条脱敏 trace 事件 |

| GET | `/api/trace/export?limit=500` | 导出事件（含 `exportedAt`、`filters`、`summary`、`timeline`） |

| GET | `/api/trace/replay?limit=100` | **审计回放**：默认过滤 `model_call` 等噪声，仅保留链路事件 |

### 回放/导出查询参数

| 参数 | 说明 |
| --- | --- |
| `limit` | 返回条数上限（replay 默认 100、最大 500；export 默认 500、最大 2000） |
| `runId` | 按 Run ID 过滤（有 scope 过滤时全文件扫描） |
| `sessionId` | 按会话 ID 过滤 |
| `taskId` | 按任务 ID 过滤 |
| `toolCallId` | 按工具调用 ID 串联 `agent_tool` / `tool_audit` / `task_step` |
| `type` | 单事件类型 |
| `types` | 逗号分隔多类型，如 `tool_audit,agent_tool` |
| `category` | 分类：`run` / `model` / `tool` / `agent` / `task` / `background` / `subagent` / `scheduler` |
| `replayOnly` | 是否仅保留回放白名单（默认 `true`；export 可设 `false` 导出全类型） |

回放白名单含：`run_start` / `run_end`、`agent_decision`、`agent_model_turn`、`run_usage_summary`、`task_status_change`、`tool_audit`、`agent_tool`、`task_step`、后台/子 Agent/调度相关事件。

响应除 `events` / `count` / `redacted` 外，还提供 `filters`、`summary`（类型计数与关联 id）、`timeline`（与运行报告相同结构）。

`capabilities.traceReplay: true` 表示已注册回放接口；`capabilities.traceReplayFilters: true` 表示 replay/export 支持上述过滤参数；`capabilities.agentDecisionTrace: true` 表示回放时间线含 Agent 决策事件；`capabilities.taskStatusTrace: true` 表示 TaskRunner 状态变化会进入 trace；`capabilities.toolCallTrace: true` 表示工具调用链路可用 `toolCallId` 串联；`capabilities.modelUsageTrace: true` 表示模型用量与运行摘要会进入 trace。修改 API 后需重启 `npm run serve`。
