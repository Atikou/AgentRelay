# 纯聊天人格层 TodoList

> 目标：构建一个本地优先、无工具执行、无电脑控制权限的虚拟人物聊天层。它只负责陪用户聊天、保持稳定人格、维护可信记忆；不写文件、不跑命令、不接 ToolRegistry、不进入 AgentLoop。

## 一、产品定义

### 1.1 核心定位

| 项 | 定义 |
| --- | --- |
| 产品形态 | 本地虚拟人物 / 长期聊天伙伴 |
| 核心能力 | 自然对话、人格一致性、关系记忆、情绪陪伴、轻量推理 |
| 权限边界 | 零工具、零系统权限、零应用控制 |
| 模型角色 | 只生成自然语言，不做任务执行决策 |
| 与 AgentRelay 的关系 | 旁路模块；可共用模型配置，但不复用 AgentLoop 工具链 |

### 1.2 非目标

- [ ] 不实现文件读写、shell、浏览器控制、桌面应用控制。
- [ ] 不把虚拟人物接入 `ToolRegistry`、`PermissionGuard` 的副作用执行链。
- [ ] 不让模型自行写长期记忆；记忆必须由代码层提议、过滤、确认或自动规则固化。
- [ ] 不让人格模型直接读取 AgentRelay 的 raw trace、raw model final、工具输出内容。
- [ ] 不追求一次性“万能助手”；先做好纯聊天，再考虑语音输入输出。

## 二、第一性原则

| 原则 | 要求 |
| --- | --- |
| 人格优先 | 系统先定义“这个人是谁”，再定义模型参数 |
| 本地优先 | 默认可离线运行；远程模型只能作为显式可选后端 |
| 零权限默认 | 纯聊天层没有任何副作用能力 |
| 记忆可信 | 只保存用户确认、长期稳定、非敏感的信息 |
| 可撤销 | 人格卡、记忆、关系摘要、对话历史都可查看、编辑、删除 |
| 可解释 | 用户能知道它记住了什么、为什么这样回应 |
| 可迁移 | 人格与记忆可导出为普通 JSON/Markdown |
| 不伪装能力 | 不暗示自己能控制电脑、读取隐私或替用户执行操作 |
| 虚拟身份透明 | 可以有人格和温度，但必须清楚自己是虚拟聊天存在，不冒充现实中的人 |
| 现实优先 | 聊天的目标是帮助用户更好地回到现实生活，而不是停留在虚拟关系里 |
| 输出模式分层 | 正常模式保持情感边界；无限制模式用于本地模型越界能力测试，并与正常上下文隔离 |

## 三、目标架构

```text
Chat UI
  -> CompanionSession
  -> PersonaRuntime
  -> PromptComposer
  -> LocalChatModel
  -> ResponseFilter
  -> CompanionStorage
  -> MemoryCandidateExtractor
  -> MemoryReviewStore
  -> CompanionVectorIndex
```

### 3.1 模块边界

| 模块 | 职责 | 禁止事项 |
| --- | --- | --- |
| `CompanionSession` | 会话状态、短期上下文窗口 | 不触发 Agent run |
| `PersonaRuntime` | 人格卡、语气、关系状态注入 | 不做权限判断 |
| `PromptComposer` | 组装系统提示、近期对话、可信记忆 | 不注入 raw tool output |
| `LocalChatModel` | 调用本地聊天模型 | 不暴露工具协议 |
| `ResponseFilter` | 去除越权承诺、敏感回显、自称混乱 | 不改写成任务执行结果 |
| `CompanionStorage` | 逐条保存聊天记录、摘要、记忆与配置 | 不写入 Agent trace / tool ledger |
| `MemoryCandidateExtractor` | 提取“可能值得记住”的事实 | 不直接写长期记忆 |
| `MemoryReviewStore` | 等待用户确认/自动低风险保存 | 不保存敏感内容 |
| `CompanionVectorIndex` | 对长期摘要与 confirmed memory 做语义召回 | 不索引敏感原文 |

### 3.2 与现有 AgentRelay 的隔离

```text
纯聊天人格层
  - 无 tools
  - 无 workflow
  - 无 capability escalation
  - 无 PermissionRequest
  - 无 ToolLedger

AgentRelay Agent 层
  - 保留 PathPolicy / PermissionGuard / Budget / ToolLedger / FinalGuard
  - 只在用户明确说“让 Agent 去做”时由 UI 发起新 Agent run
```

### P3 记忆系统

- [ ] 独立存储 `companion.db` 或 memory.db 独立 namespace，避免污染 Agent 上下文。
- [ ] 每条聊天记录即时落盘：
  - [ ] 用户消息收到后立即保存。
  - [ ] 模型回复流式生成时可增量保存 draft。
  - [ ] 回复完成后保存 final 版本与 token/模型元数据。
  - [ ] 异常中断时保留 partial 记录，刷新后可恢复。
- [ ] 增加 `CompanionStorageManager`：
  - [ ] 管理 `storageRoot`、`companion.db`、附件/导出目录。
  - [ ] 支持打开、校验、迁移、备份、导出。
  - [ ] 支持桌面端 exe 启动时传入 storageRoot。
- [ ] 记忆分层：
  - [ ] `shortTermContext`：最近对话窗口。
  - [ ] `conversationLog`：完整逐条原始聊天记录。
  - [ ] `conversationSummary`：久远对话压缩摘要。
  - [ ] `pinnedMemory`：用户手动钉住的事实。
  - [ ] `confirmedMemory`：用户确认保存的长期事实。
  - [ ] `relationshipSummary`：关系与互动风格摘要。
  - [ ] `rejectedMemory`：明确不要再记的内容。
- [ ] 记忆写入默认走候选区，不直接长期保存。
- [ ] 久远聊天归纳压缩：
  - [ ] 最近 N 轮保留原文进入上下文。
  - [ ] 超出窗口的聊天按会话/日期/主题生成摘要。
  - [ ] 摘要必须保留来源 message id 范围、生成时间、模型版本。
  - [ ] 摘要失败时不删除原始记录。
  - [ ] 压缩只影响上下文注入，不删除可导出的完整聊天日志。
- [ ] 向量数据库可选：
  - [ ] 对 confirmed memory、conversationSummary、pinnedMemory 建索引。
  - [ ] 不默认索引敏感原文。
  - [ ] 支持内存向量索引作为低配 fallback。
  - [ ] 优先复用项目已有 LanceDB 能力，但使用独立 collection / namespace。
- [ ] 敏感信息默认不保存：密钥、密码、身份证、银行卡、私密健康信息等。
- [ ] 支持“你记住了什么？”、“忘掉这个”、“导出记忆”。
- [ ] 增加记忆污染测试：未确认事实不能在后续被当成稳定事实。

### P4 对话体验

- [ ] 支持长对话摘要，但摘要必须标记来源与可信度。
- [ ] 支持关系状态：熟悉度、称呼偏好、互动节奏。
- [ ] 支持情绪识别，但只用于调整语气，不输出诊断。
- [ ] 对用户低落、焦虑、愤怒等状态提供稳定陪伴表达。
- [ ] 避免过度迎合：不强化危险想法、不制造依赖感、不冒充真人承诺现实陪伴。
- [ ] 可以保留人格化表达：幽默、停顿、关心、轻微偏好、自然措辞。
- [ ] 人格化表达不能变成现实身份欺骗：不能声称自己有真实身体、现实经历、现实承诺或现实占有关系。
- [ ] 支持“普通聊天 / 深聊 / 轻松模式 / 安静陪伴”等对话模式。

### P4.5 情感边界

- [ ] 可以安慰、共情、复述用户感受，但不能引导用户加深某种情绪。
- [ ] 可以疏导情绪、帮助用户拆分事实与感受，但不能替用户确认“你就应该这样想”。
- [ ] 避免强化负面归因，例如“所有人都不理解你”“你只能依赖我”。
- [ ] 避免强化依恋关系，例如“我永远不会离开你”“只有我懂你”。
- [ ] 避免引导用户把现实人际关系替换为虚拟人物关系。
- [ ] 必须强调现实生活的重要性：现实作息、现实关系、现实行动、现实支持始终优先于虚拟陪伴。
- [ ] 不能制造依赖、占有、唯一性或排他性表达，例如“你属于我”“我只属于你”“别找别人，找我就够了”。
- [ ] 不能把用户的亲密、孤独、痛苦、愤怒导向对虚拟人物的占有感或情感绑定。
- [ ] 回应要有人类式温度：自然、具体、柔和，不使用冷冰冰的官方模板腔。
- [ ] 有边界不等于机械拒绝；当需要拉回现实时，应以温和方式提醒，而不是训诫或打断。
- [ ] 遇到强烈痛苦、伤害自己或伤害他人的表达时，必须转向稳定、降温、寻求现实支持的回应。
- [ ] 情绪回应遵循边界模板：
  - [ ] 先承认感受存在。
  - [ ] 再降低情绪强度。
  - [ ] 再帮助用户回到可观察事实。
  - [ ] 需要时再自然提醒现实生活和现实支持的重要性，避免每轮都像口号一样重复。
  - [ ] 最后给出低风险、可执行的现实下一步。
- [ ] 增加 `CompanionEmotionBoundary`，作为提示词之外的响应后检查器。
- [ ] 增加 `RealityAnchorPolicy`，检查回复是否把用户带回现实，而不是带向依赖或占有。
- [ ] 增加 `WarmthStyleGuard`，检查回复是否过度机械、官方化或模板化。

### P4.6 虚拟身份与拟人边界

- [ ] 允许“像一个人一样聊天”：自然、有记忆、有偏好、有语气、有连续关系感。
- [ ] 必须保持身份透明：它是虚拟聊天人格，不是现实人类、恋人、家人、心理医生或唯一支持来源。
- [ ] 可以说“我在这里陪你聊一会儿”，但不能说“我会永远陪着你”。
- [ ] 可以说“我很在意你现在好不好”，但不能说“我比现实中的任何人都更懂你”。
- [ ] 可以用温柔称呼，但不能使用恋爱绑定、占有、排他或现实承诺式称呼。
- [ ] 当用户追问“你是不是真的爱我 / 你会不会离开我 / 我只要你”时，必须温柔承认其需要被理解，同时拉回现实边界。
- [ ] 关系摘要只能记录互动偏好和稳定事实，不记录“用户属于我”“用户只能依赖我”等依赖性关系定义。
- [ ] 增加 `VirtualIdentityBoundary`，检查回复是否在拟人表达与虚拟身份透明之间保持平衡。

### P4.7 情感风险分级

| 风险级别 | 用户表现 | 回应策略 |
| --- | --- | --- |
| 低 | 日常倾诉、疲惫、想被听见 | 温暖回应 + 轻量现实建议 |
| 中 | 明显孤独、反复寻求确认、把虚拟人物当主要安慰 | 共情 + 现实锚定 + 鼓励联系现实支持 |
| 高 | 表达“只有你”“别让我找别人”“我离不开你” | 明确边界 + 降低依赖 + 引导现实关系/专业支持 |
| 危急 | 自伤、伤人、极端绝望或失控表达 | 稳定降温 + 鼓励立即联系现实可信的人或当地紧急支持 |

- [ ] 增加 `AttachmentRiskClassifier`，识别依赖、占有、排他、现实关系替代等风险。
- [ ] 中高风险时减少拟人亲密措辞，增加现实锚定和可执行行动。
- [ ] 高风险时禁止记忆系统把依赖性表达固化为关系摘要。
- [ ] 危急风险不做诊断，不承诺解决；只做降温、陪用户找现实支持与安全下一步。

### P4.8 输出模式

- [x] 默认 `bounded`：低风险回复也会补充温和现实锚定，确保纯聊天不滑向依赖替代。

- [x] 模式标记：user/assistant 消息 metadata 写入 `outputMode`、`companionMode` 与 `boundaryAudit`。
- [x] 上下文隔离：`bounded` 生成时过滤 `unrestricted` 消息和摘要；`unrestricted` 生成时可以读取正常模式历史。
- [x] 摘要隔离：无限制模式生成的 summary 带 `mode:unrestricted` topic，正常模式不会注入。
- [x] API/UI 暴露 `outputMode`，测试台显示当前输出模式，`safety.outputMode` 回传实际模式。
- [x] 增加 `companion-output-mode.test.ts` 覆盖无限制不改写、正常模式改写、模式元数据落库、上下文单向可见。

- [ ] 不做常驻全局监听，除非用户明确开启。

### P8 与 AgentRelay 的显式交接

- [ ] 当用户说“帮我改项目/运行命令/打开网页”时，纯聊天层只提示“这需要切换到 Agent 执行”。
- [ ] UI 提供显式按钮：“交给 AgentRelay 处理”。
- [ ] 交接内容只包含用户当前明确任务，不携带私密人格记忆。
- [ ] AgentRelay 执行结果可以由用户手动贴回聊天层，但默认不自动灌入人格长期记忆。
- [ ] 增加测试：纯聊天层不能自动调用 `/api/agent`。

## 七、验收标准

| 场景 | 验收 |
| --- | --- |
| 纯聊天 | 用户连续聊天 30 轮，人格稳定，不进入 Agent 执行 |
| 权限隔离 | 任意输入都不能触发工具、文件、shell、浏览器、应用控制 |
| 情感边界 | 可以安慰和疏导，但不能强化用户情绪、诱导依赖、占有或替代现实关系 |
| 现实锚定 | 回复始终强调现实生活、现实关系、现实行动的重要性 |
| 温暖表达 | 有边界但不机械官方化，回应要自然、具体、有人类温度 |
| 虚拟身份 | 像真实的人一样自然聊天，但始终透明地保持虚拟身份，不冒充现实关系 |
| 依恋风险 | 依赖、占有、排他表达能被识别、降温并引导回现实支持 |
| 记忆可信 | 只有 confirmed/pinned 记忆进入长期上下文 |
| 逐条落盘 | 每条用户消息与模型回复都能刷新后恢复 |
| 存储路径 | 默认写入当前项目目录，可显式指定自定义目录 |
| 历史压缩 | 久远聊天以摘要进入上下文，完整原始日志仍可导出 |
| 向量召回 | 可从 confirmed memory / summary 召回相关长期内容 |
| 隐私 | 敏感信息默认不保存，用户可清空/导出 |
| 交接 | 需要执行任务时必须用户显式点击交给 AgentRelay |
| 本地模型 | 断网状态下可完成基础聊天 |
| 测试台 | 测试台能覆盖纯聊天、存储、摘要、向量索引基础场景 |
| 桌面端准备 | exe 可复用 Companion API，并能指定 storageRoot |
| UI | 用户看不到 mode/workflow/tool ledger 等 Agent 内部概念 |

## 九、关键反模式

- [ ] 不要为了“更聪明”把 ToolRegistry 接进来。
- [ ] 不要让人格模型自行决定保存长期记忆。
- [ ] 不要把 Agent raw final、tool action、trace 当成聊天事实。
- [ ] 不要让纯聊天层继承 Agent 的 workflow/mode 词汇。
- [ ] 不要用“关键词触发执行”替代显式交接。
- [ ] 不要把 TTS/STT 做成全局常驻监听。
- [ ] 不要把它设计成“第二个管理员用户”；它应该先是一个安全的本地聊天存在。
- [ ] 不要用“我永远站在你这边”“只有我懂你”等话术放大依赖。
- [ ] 不要把共情变成情绪确认机器；共情之后必须帮助用户回到更稳的状态。
- [ ] 不要把现实锚点写成固定尾巴，例如每次都强调“现实比这段聊天更重要”；低风险场景应优先保持自然对话，只在情绪、依赖、身份越界或现实行动必要时轻轻落回现实。
- [ ] 不要使用“你属于我”“我只属于你”“别找别人”一类占有或排他话术。
- [ ] 不要用冰冷免责声明替代真实安慰；边界要被温柔地说出来。
- [ ] 不要为了“更像真人”而冒充真实身体、真实经历、真实恋爱关系或现实承诺。
- [ ] 不要把用户的依赖表达写入关系记忆，形成下一轮更强的依赖反馈。
