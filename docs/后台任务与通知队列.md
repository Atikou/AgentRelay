# 后台任务与通知队列（M4）

AgentRelay 支持将长时间 Shell 命令放到**后台**运行，并在任务完成时通过**通知队列**异步告知主 Agent。主 Agent 仅在**安全点**消费通知，避免在写文件或执行工具中途被打断。

## 能力概览

| 能力 | 说明 |
| --- | --- |
| 后台启动 | `spawn` + `shell`，不阻塞 HTTP / 主循环 |
| 输出记录 | 累积 stdout / stderr（单流上限 512KB，保留尾部） |
| 状态查询 | `running` / `completed` / `failed` / `cancelled` / `timed_out` |
| 取消任务 | Windows 使用 `taskkill /T /F` 终止进程树 |
| 命令超时 | 启动时可选 `timeoutMs`（1s–24h）；到期终止进程树，状态 `timed_out`；未设置则不自动超时 |
| **输出匹配** | `outputRules` 对 stdout/stderr 子串/正则匹配（错误关键字、测试完成、ready 日志等） |
| **触发下一步** | `triggerOnMatch` 规则命中后自动 `executeUnattendedTrigger`（无人值守 Agent） |
| 通知队列 | 任务结束时写入；JSONL 持久化到 `data/notifications/` |
| 通知排序与去重 | `priority` 排序；`dedupeKey` 原地更新同一事件；`mergeKey` 将多条未消费通知折叠为一条（`mergeCount` 递增） |
| 安全点消费 | AgentLoop 在「循环开始」与「每次工具完成后」drain 通知 |

## HTTP 接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `POST` | `/api/background/start` | 见下方请求体 |
| `GET` | `/api/background` | 列出全部后台任务 |
| `GET` | `/api/background/:id` | 查询单个任务（含输出） |
| `POST` | `/api/background/:id/cancel` | 取消运行中的任务 |
| `GET` | `/api/notifications?pending=1` | 列出通知（`pending=1` 仅未消费） |
| `POST` | `/api/notifications/consume` | 手动在安全点消费全部待处理通知 |

`POST /api/agent` 的智能体循环已接入通知队列：返回体中的 `notifications` 字段为本轮在安全点已消费的通知列表。

### `POST /api/background/start` 请求体（节选）

```json
{
  "command": "npm test",
  "timeoutMs": 600000,
  "outputRules": [
    { "name": "test_passed", "pattern": "passed", "stream": "stdout", "ignoreCase": true },
    { "name": "npm_error", "pattern": "npm ERR!", "stream": "stderr" }
  ],
  "triggerOnMatch": {
    "goal": "根据测试输出汇总结果并建议修复",
    "mode": "any",
    "requireSuccess": true
  }
}
```

| 字段 | 说明 |
| --- | --- |
| `outputRules[].name` | 规则名（写入 `outputMatches`） |
| `pattern` / `regex` / `ignoreCase` | 子串或正则；可选 `stream`: stdout / stderr / both |
| `fireOnStream` | `true` 时进程未结束即可命中（适合 `listening on` ready 日志） |
| `triggerOnMatch.goal` | 命中后交给 `Orchestrator.executeUnattendedTrigger` 的 Agent 目标 |
| `triggerOnMatch.mode` | `all`（默认）全部规则命中；`any` 任一命中 |
| `requireSuccess` | 默认 `true`：仅 `exitCode=0` 且 `completed` 时触发 |

任务完成通知 `payload` 含 `outputMatches`、`matchedRules`、`triggeredRunId`。

内置预设见 `OUTPUT_MATCH_PRESETS`（`outputMatcher.ts`）：`npmError`、`testPassed`、`serverReady`。

## 通知元数据

`AgentNotification` 当前包含 `priority`（`low` / `normal` / `high`）、`runId`、`dedupeKey` 与 `mergeKey`。

| 键 | 行为 |
| --- | --- |
| `dedupeKey` | 同一未消费通知原地更新（保留 id），适合防抖后的重复事件 |
| `mergeKey` | 不同 `dedupeKey` 的多条通知折叠为一条，`payload.mergeCount` 递增，`mergedMessages` 保留最近 5 条摘要 |

历史 JSONL 无 `priority` 时按 `normal` 回放。调度器通知带 `mergeKey: scheduler:{triggerId}`；若创建了 `scheduled` Run，会在 `runId` 与 `payload.runId` 中写入。

## 安全点定义

与实现指南一致，当前在以下位置消费通知：

1. **AgentLoop 每次 `run()` 开始前**（承接上一轮遗留通知）
2. **每个工具步骤执行完成后**（下一轮模型调用前）
3. **测试台「通知队列 → 手动消费」**（调试 / 非 Agent 场景）

不在模型推理中途、工具执行中途注入通知。

## 测试台

启动 `npm run serve` 后，左侧：

- **后台任务**：输入命令后台启动、刷新列表、取消运行中任务
- **通知队列**：查看待处理通知、手动消费

## 安全边界

- 后台任务默认只直接启动 `safe` 命令。
- `caution` 命令（如安装依赖、删除/移动/覆盖类命令）未传 `confirm:true` 时返回 `needsConfirmation` 与 `risk.reason`，不创建后台任务。
- `dangerous` 命令直接拒绝，仍由 `BackgroundTaskManager` 做二次风险拦截。
- 主 Agent 消费后台/调度通知时会按不可信外部数据处理，通知中出现可疑 prompt injection 片段会被围栏标记。

## 代码位置

```text
agent-relay/src/background/
├─ BackgroundTaskManager.ts   # spawn、输出、取消、完成时 enqueue、输出匹配触发
├─ outputMatcher.ts           # 规则评估与 trigger 判定
├─ NotificationQueue.ts       # JSONL 持久化 + drain
├─ types.ts
└─ index.ts
```

自检：`npm run test:background`（10 项）、`npm run test:output-matcher`（5 项）。

## 与 shell_run 的区别

| | `shell_run` 工具 | 后台任务 API |
| --- | --- | --- |
| 阻塞 | 同步等待命令结束（默认 30s，可配至 5min） | 立即返回 task id |
| 超时 | 工具层 `timeoutMs` 到期即失败 | 可选 `timeoutMs`，到期 `timed_out` |
| 适用 | 短命令、Agent 需要即时结果 | `npm run dev`、`build`、长时间测试 |
| 通知 | 无 | 完成时写入通知队列 |
