# 数据存储边界

本文说明 AgentRelay 本地数据目录 `agent-relay/data/` 下 **SQLite / JSONL / LanceDB / 文件** 的职责划分，便于排错、备份与后续迁移。

> Schema 版本与迁移审计见 `src/storage/sqliteMigration.ts`；`GET /api/config.schemaVersions` 可探测 `memory.db`（v8）/ `tools.db` 版本。

---

## 1. 总览

| 类型 | 用途 | 是否可查询业务状态 | 典型消费方 |
|------|------|-------------------|------------|
| **SQLite** | 结构化业务状态、路由/工具审计表 | ✅ 是（SQL / Store） | ContextManager、PlanStore、RouteLogStore、ToolStorage |
| **JSONL** | 追加式流水 / 事件日志 / 队列回放 | ⚠️ 部分（内存重建或顺序扫描） | TraceLogger、NotificationQueue、Scheduler |
| **LanceDB** | 向量检索（语义记忆） | ✅ 是（向量 API） | SemanticRetriever |
| **普通文件** | 备份、会话镜像、错误旁路 | 视路径而定 | ToolStorage.backups、ContextManager 日志 |

**记忆口诀**：需要 **按 id 查、改状态、JOIN** → SQLite；需要 **只追加、顺序复盘、崩溃可重放** → JSONL。

## 2. 目录树

```text
agent-relay/data/
├─ traces/
│  └─ trace.jsonl                 # 全局审计 trace（脱敏）
├─ notifications/
│  └─ notifications.jsonl        # 通知队列流水（enqueue + consume 记录）
├─ scheduler/
│  └─ triggers.jsonl             # 触发器定义与状态变更流水
└─ agent_data/
   ├─ memory.db                   # 主库（v7）：会话/任务/计划/Run/路由表
   ├─ tools.db                    # 工具库（v1）：tool_logs / file_changes / backups
   ├─ lancedb/                    # 向量表（语义检索）
   ├─ backups/                    # 工作区文件备份（按 changeId / 日期）
   ├─ files/                      # 上下文附件等
   └─ logs/
      ├─ messages/{sessionId}.jsonl   # 会话消息镜像（旁路，非主数据源）
      └─ context-errors.jsonl         # 上下文压缩/持久化错误
```

## 3. SQLite：可查询业务状态

### 3.1 `memory.db`（`DatabaseManager`，schema v8）

| 表 / 区域 | 业务含义 | 写入模块 |
|-----------|----------|----------|
| `sessions` / `messages` | 对话会话与消息（**主数据**） | `MessageStore`、`AgentLoop` 持久化 |
| `conversation_summaries` | 会话摘要 | `SummaryStore`、压缩流程 |
| `memories` + FTS | 长期记忆与全文检索 | `MemoryStore`、`MemoryExtractor` |
| `tasks` / `task_steps` / `task_step_dependencies` / `task_attempts` | 任务执行状态机 | `TaskStore`、`TaskRunner` |
| `runs` | 统一 Run（Agent/Task/Chat/Plan…） | `RunStore`、`Orchestrator` |
| `task_plans` / `task_plan_versions` / `task_plan_previews` / `task_plan_approvals` / `task_plan_runs` | 内部可执行计划与审批 | `PlanStore`、`PlanService` |
| `user_visible_plans` | 用户可见计划 Markdown/Todo | `PlanStore` |
| `projects` | 项目元数据 | `ProjectStore` |
| `model_route_logs` / `model_call_logs` / `model_collaboration_runs` / `fallback_logs` | 模型路由与协作审计 | `RouteLogStore` 等（同库） |
| `model_eval_runs` / `model_eval_results` | V7 离线路由评测结果 | `ModelEvalStore` / `EvalSetRunner` |
| `schema_migrations` | 迁移审计 | `sqliteMigration` |
| `messages_fts` / `summaries_fts` / `memories_fts` | FTS5 虚拟表 | `DatabaseManager` |

**查询 API 示例**：`GET /api/runs`、`GET /api/routing/logs`、`GET /api/context/*`、计划/任务相关 HTTP。

### 3.2 `tools.db`（`ToolStorage`，schema v1）

| 表 | 业务含义 |
|----|----------|
| `tool_logs` | 工具调用审计（脱敏后落盘） |
| `file_changes` | 写操作变更记录（changeId 回滚链） |
| `backups` / `backup_files` | 备份批次元数据 |
| `schema_migrations` | 迁移审计 |

实际备份文件在 `agent_data/backups/` 目录，表内只存路径与 hash。

## 4. JSONL：审计流水与队列回放

JSONL 文件 **只追加**（或整条重写 replay 状态），不适合复杂查询；启动时由对应模块 **replay 进内存** 或 **顺序读取尾部**。

| 文件 | 内容类型 | 是否业务主数据 | 说明 |
|------|----------|----------------|------|
| `traces/trace.jsonl` | **审计 trace** | ❌ 旁路 | `TraceLogger`：agent_decision、tool_audit、task_status_change 等；`GET /api/trace/*` |
| `notifications/notifications.jsonl` | **队列流水** | ⚠️ 内存为主 | 每行 `enqueue` 或 `consume`；`NotificationQueue` 启动 replay；未消费通知在内存 Map |
| `scheduler/triggers.jsonl` | **触发器流水** | ⚠️ 内存为主 | 触发器 CRUD / 状态；`Scheduler` replay 后驻内存 |
| `logs/messages/{sessionId}.jsonl` | **镜像日志** | ❌ 旁路 | 会话消息副本，主数据仍在 `messages` 表 |
| `logs/context-errors.jsonl` | **错误旁路** | ❌ 排错用 | 上下文持久化失败记录 |

### 与 SQLite 的重复边界

| 领域 | SQLite（权威） | JSONL（辅助） |
|------|----------------|---------------|
| 对话消息 | `messages` 表 | 可选 `logs/messages/*.jsonl` 镜像 |
| 工具审计 | `tool_logs` 表（结构化） | `trace.jsonl` 中 `tool_audit` 事件（时间线） |
| 模型调用 | `model_call_logs` 表 | `trace.jsonl` 中 `agent_model_turn` 等 |
| 任务状态 | `task_steps.status`、`runs.status` | `trace.jsonl` 中 `task_status_change` |
| 后台长任务 | **仅进程内** `BackgroundTaskManager` Map | 无持久化（重启丢失） |

## 5. LanceDB 与文件

| 路径 | 用途 |
|------|------|
| `agent_data/lancedb/` | 消息/摘要向量；`SemanticRetriever` 与 FTS 多路合并 |
| `agent_data/backups/` | `backup_file` / `rollback_change` 实体文件 |
| `agent_data/files/` | 上下文包附件等 |

向量与 `memories` / `messages` 表 **配合使用**：结构化字段在 SQLite，embedding 在 LanceDB。

---

## 6. 备份与迁移建议

1. **业务恢复**：优先备份整个 `data/agent_data/`（含 `memory.db`、`tools.db`、`lancedb/`）。
2. **审计复盘**：另备 `data/traces/trace.jsonl`；路由细节还可查 `memory.db` 内路由表。
3. **队列状态**：`notifications.jsonl`、`triggers.jsonl` 可随 `data/` 一并备份；单独拷贝时需与进程停机配合避免半行写入。
4. **schema 升级**：改表必须走 `memoryDbMigrations.ts` / `toolsDbMigrations.ts` 递增版本，见 `AGENTS.md` · SQLite 迁移约定。

## 7. 相关 HTTP

| 接口 | 数据源 |
|------|--------|
| `GET /api/config.schemaVersions` | SQLite `schema_migrations` |
| `GET /api/runs` / `GET /api/runs/:id/report` | `runs` 表 + `trace.jsonl` |
| `GET /api/routing/logs` | `model_*` / `fallback_logs` 表 |
| `GET /api/trace/recent|export|replay` | `trace.jsonl` |
| `GET /api/notifications/*` | 内存队列（持久化在 JSONL） |
| `GET /api/scheduler/*` | 内存触发器（持久化在 JSONL） |

## 8. 后续待统一（已知缺口）

- `BackgroundTaskManager` 任务记录 **未持久化**（仅内存）。
- 部分 trace 事件与 SQLite 审计表 **双写**（按时间线 vs 按关系查询），暂无自动对账。
- 会话 `logs/messages/*.jsonl` 与 `messages` 表 **可能重复**，以 SQLite 为准。
